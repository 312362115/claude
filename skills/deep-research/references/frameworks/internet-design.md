# 设计职能分析框架

> 调研涉及用户体验、设计评审、交互优化等命题时，叠加以下框架。

## 核心框架

### HEART 框架（Google UX 度量）

- **适用场景**：大规模产品 UX 度量、设计决策量化、AB 测试指标选取
- **操作步骤**（GSM 流程）：
  1. **Goals**：确定要度量的 UX 目标
  2. **Signals**：找到反映目标达成的用户行为信号
  3. **Metrics**：将信号量化为可追踪的指标
- **产出格式**：HEART 五维度得分表 + 趋势对比

| 维度 | 定义 | 指标示例 |
|------|------|---------|
| Happiness | 用户主观态度 | NPS、CSAT、SUS |
| Engagement | 互动深度 | DAU/MAU、人均停留时长 |
| Adoption | 新用户/功能采纳 | 功能采纳率、7 日新用户数 |
| Retention | 持续使用 | 次日/7 日/30 日留存率 |
| Task Success | 任务完成 | 完成率、错误率、Time on Task |

- **常见误用**：五个维度全做导致指标过多；不设基线无法衡量改进

### Nielsen 十大可用性启发式

- **适用场景**：设计评审、可用性审查、竞品 UX 分析
- **操作步骤**：
  1. 选择 3-5 名评估者独立评估
  2. 每人逐一检查 10 条启发式
  3. 每个问题评定严重等级（0=不是问题，1=外观，2=次要，3=重要，4=灾难）
  4. 合并去重，按严重等级排序
  5. 输出评估报告 + 修复建议
- **产出格式**：问题清单（问题 | 违反原则 | 严重等级 | 截图 | 修复建议）

十条原则速查：系统状态可见 | 匹配现实世界 | 用户控制自由 | 一致性标准 | 错误预防 | 识别优于回忆 | 灵活高效 | 美学极简 | 帮助识别恢复错误 | 帮助文档

- **常见误用**：全部标为"重要"没有分级；不附截图证据

### Design Sprint（Google Ventures）

- **适用场景**：新产品/新功能快速验证、创新探索
- **操作步骤**：
  1. Day 1 - Map：理解问题、确定目标
  2. Day 2 - Sketch：发散方案、各自出方案
  3. Day 3 - Decide：投票选方案、细化
  4. Day 4 - Prototype：搭建可测试原型
  5. Day 5 - Test：5 个用户测试 + 收集反馈
- **产出格式**：测试报告 + 可行性判断 + 下一步建议

### 选择逻辑

| 场景 | 推荐框架 |
|------|---------|
| 大规模 UX 度量 | HEART |
| 设计质量评审 | Nielsen 启发式 |
| 新方向快速验证 | Design Sprint |

## 典型报告骨架

```
1. 执行摘要（审计范围 + 方法论 + Top 5 问题）
2. 评估方法（评估者 + 框架 + 流程）
3. 详细发现（问题清单按严重等级排序 + 正面发现）
4. 体验度量（HEART 各维度得分 + 与上版本/竞品对比）
5. 优先级排序（影响-工作量矩阵 + 修复优先级）
6. 行动计划（Quick Wins + 中期改善 + 长期优化）
```

## 数据采集指南

### 内部数据（优先）
- 可用性测试：原型/上线验证（Maze/UserTesting）
- 行为数据：热力图+录屏（Hotjar/FullStory）
- A/B 测试：设计方案对比（Optimizely/VWO）

### 外部数据
- 设计规范：Apple HIG、Material Design、Ant Design
- 竞品截图：UI 趋势、交互模式（Mobbin/Dribbble）
- UX 基准：行业平均指标（Baymard Institute/Nielsen Norman Group）

## 常见反模式

- **像素完美陷阱**：过度关注视觉细节忽视用户任务流
- **设计不做度量**：只凭感觉评价设计好坏
- **专家偏见**：设计师评审自己的设计容易盲点
- **测试样本过小**：5 人测试发现 85% 问题但不能替代定量验证
- **美学优先**：关注视觉美感忽视可用性

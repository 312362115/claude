# 技术职能分析框架

> 调研涉及架构决策、技术选型、性能分析等命题时，叠加以下框架。

## 核心框架

### ATAM（架构权衡分析法）

- **适用场景**：重大架构决策、系统重构评审
- **操作步骤**：
  1. 介绍业务驱动因素和当前架构
  2. 识别架构方法，生成质量属性效用树（性能/可用性/安全/可维护/可扩展）
  3. 分析敏感点（影响单个质量属性）和权衡点（影响多个质量属性）
  4. 头脑风暴场景并优先排序，第二轮分析
  5. 输出权衡矩阵 + 风险清单
- **产出格式**：质量属性效用树 + 权衡矩阵 + 敏感点/风险清单
- **常见误用**：只关注功能需求忽视非功能需求；不做质量属性量化

### C4 模型（架构可视化）

- **适用场景**：架构文档编写、团队沟通、新人上手
- **操作步骤**：
  1. **Context**：画系统与外部角色/系统的关系图
  2. **Container**：展开为应用、数据库、消息队列等
  3. **Component**：展开容器内的模块和交互
  4. **Code**：类图级别（通常由 IDE 自动生成，按需画）
- **产出格式**：四层递进的架构图（按受众选择展开到哪层）

### 技术选型决策矩阵

- **适用场景**：技术栈选择、中间件选型、框架对比
- **操作步骤**：
  1. 确定评估维度和权重：功能匹配度(25%) / 性能(20%) / 生态成熟度(20%) / 团队适配(15%) / 运维成本(10%) / 许可合规(10%)
  2. 每个候选方案逐维度评分（1-5 分）
  3. 加权求和排序，差距小的需 PoC 验证
- **产出格式**：决策矩阵表（方案 × 维度 × 加权分）+ PoC 验证结果
- **常见误用**：只看文档不做 PoC；忽视运维成本和团队学习曲线

### ADR（架构决策记录）

- **适用场景**：记录每次技术决策的来龙去脉，防止知识流失
- **操作步骤**：
  1. 记录标题、状态（提议/已采纳/已废弃）
  2. 写清上下文（为什么要做这个决策）
  3. 记录决策内容和理由
  4. 列出后果（正面和负面）
- **产出格式**：`ADR-XXX: 标题 | 状态 | 上下文 | 决策 | 理由 | 后果`

### USE 方法（性能分析三板斧）

- **适用场景**：性能问题排查、容量规划
- **操作步骤**：
  1. 列出所有资源（CPU/内存/磁盘/网络/锁等）
  2. 对每个资源检查：Utilization（利用率%）、Saturation（排队等待量）、Errors（错误数）
  3. 从异常项追溯根因
- **产出格式**：资源 × USE 三维检查表 + 瓶颈定位结论

### 选择逻辑

| 场景 | 推荐框架 |
|------|---------|
| 重大架构决策 | ATAM |
| 架构文档/沟通 | C4 模型 |
| 技术栈选型 | 决策矩阵 + PoC |
| 决策记录沉淀 | ADR |
| 性能问题排查 | USE 方法 |

## 典型报告骨架

```
1. 背景与业务价值（现状 + 瓶颈 + 预期收益）
2. 需求分析（功能需求 + 非功能需求量化指标 + 约束条件）
3. 方案设计（C4 Container 图 + 核心模块 + 数据模型 + 接口设计）
4. 技术选型（决策矩阵 + PoC 验证结果 + 选型结论）
5. 质量保障（性能设计 + 高可用 + 安全 + 监控告警）
6. 实施计划（里程碑 + 风险评估 + 灰度策略）
```

## 数据采集指南

### 内部数据（优先）
- 压测/Benchmark：性能对比、容量规划（JMeter/k6/wrk）
- 线上监控：现状分析、问题定位（Prometheus+Grafana/DataDog）
- Profiling：性能瓶颈定位（pprof/async-profiler）

### 外部数据
- 官方文档：功能说明、性能基准
- GitHub：社区活跃度（Stars/Issues/Contributors 趋势）
- 技术博客/论文：实战经验（InfoQ/掘金/Medium/arXiv）

---

## 常见反模式

- **简历驱动开发**：为学新技术引入不必要复杂度
- **银弹思维**：以为一个技术能解决所有问题
- **不做 PoC**：只看文档就做决策
- **Benchmark 陷阱**：不同硬件环境的数据不可直接比较
- **忽视运维成本**：选型只看功能不看运维复杂度

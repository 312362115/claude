# 测试职能分析框架

> 调研涉及质量保障、测试策略、质量度量等命题时，叠加以下框架。

## 核心框架

### 测试金字塔

- **适用场景**：测试策略制定、自动化测试投资分配
- **操作步骤**：
  1. 评估当前测试分布（倒金字塔 = 红灯信号）
  2. 优先补充单元测试覆盖（目标占比 70%）
  3. 关键路径加集成测试（目标 20%）
  4. 核心用户旅程加 E2E 测试（目标 10%）
  5. 持续监控测试执行时间和通过率

| 层级 | 占比 | 速度 | 成本 | 典型工具 |
|------|------|------|------|---------|
| 单元测试（底） | 70% | ms 级 | 低 | Jest/JUnit/pytest |
| 集成测试（中） | 20% | 秒级 | 中 | Testcontainers/Supertest |
| E2E 测试（顶） | 10% | 分钟级 | 高 | Playwright/Cypress |

- **产出格式**：当前分布 vs 目标分布 + 改进路径
- **常见误用**：冰淇淋反模式（E2E 多单测少）；为覆盖率写无意义测试

### 敏捷测试象限

- **适用场景**：测试类型规划、测试团队能力建设
- **操作步骤**：
  1. 按四象限评估当前测试覆盖完整性
  2. 识别缺失的象限，制定补充计划
  3. 按项目阶段动态调整各象限投入

| 象限 | 面向 | 目的 | 测试类型 |
|------|------|------|---------|
| Q1（左下） | 技术 | 支持开发 | 单元测试、组件测试 |
| Q2（右下） | 业务 | 支持开发 | 功能测试、故事测试 |
| Q3（右上） | 业务 | 评估产品 | 探索性测试、UAT |
| Q4（左上） | 技术 | 评估产品 | 性能测试、安全测试 |

- **产出格式**：四象限覆盖度评估 + 投入建议

### 质量度量指标体系

- **适用场景**：质量状态监控、质量改进决策
- **操作步骤**：
  1. 搭建质量指标看板（覆盖下表核心指标）
  2. 设定基线和目标值
  3. 按趋势分析定位问题模块

| 指标 | 计算方式 | 健康标准 |
|------|---------|---------|
| 缺陷密度 | 缺陷数 / KLOC | <5（一般），<1（优秀） |
| 缺陷拦截率 | 测试发现 / 总缺陷 | >85% |
| 缺陷逃逸率 | 线上缺陷 / 总缺陷 | <15% |
| 测试覆盖率 | 已覆盖行数 / 总行数 | >80%（核心模块） |
| 自动化率 | 自动化用例 / 总用例 | >70% |
| MTTR | 修复总时长 / 缺陷数 | P0 <24h，P1 <72h |

- **产出格式**：质量指标仪表盘 + 趋势图 + 瓶颈分析
- **常见误用**：覆盖率虚高但关键分支没测到；指标游戏

### 选择逻辑

| 场景 | 推荐框架 |
|------|---------|
| 测试投资策略 | 测试金字塔 |
| 测试类型规划 | 敏捷测试象限 |
| 质量状态监控 | 质量度量指标体系 |
| DevOps 质量内建 | 左移测试（需求→设计→编码→构建→部署各阶段嵌入测试） |

## 典型报告骨架

```
1. 概述（测试范围 + 周期 + 关键结论摘要）
2. 执行总结（用例统计 + 金字塔分布 + 自动化率）
3. 缺陷分析（按严重等级 + 趋势图 + Top5 模块根因 + 密度/逃逸率）
4. 覆盖率（代码覆盖 + 需求覆盖 + 高风险模块专项）
5. 性能与安全（压测结果 + 安全扫描）
6. 质量评估（发布风险 Red/Yellow/Green + 遗留问题 + 改进建议）
```

## 数据采集指南

### 内部数据（优先）
- CI/CD 平台：构建通过率、测试时长（Jenkins/GitHub Actions）
- 缺陷管理：历史缺陷数据（Jira/Tapd/禅道）
- 线上监控：异常、崩溃率（Sentry/Firebase Crashlytics）

### 外部数据
- 行业基准：DORA Metrics、Accelerate 报告
- 工具对比：各测试框架官方文档和 benchmark

## 常见反模式

- **测试后置**：开发完成才开始测试，修复成本高
- **脆弱测试**：E2E 频繁因环境问题失败，团队忽略红灯
- **手动回归**：每次发版全量手动用例，效率极低
- **文档和用例脱节**：测试用例不随需求变更更新
- **只关注功能测试**：忽视性能、安全、兼容性

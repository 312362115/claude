# 决策复盘卡（Decision Postmortem）

> 对应 framework：`frameworks/self-improvement-loop.md` §L3 错判归因
>
> **两用模板**：错判归因 + 成功复盘（成功也要归因，否则下次复制不了）
>
> **存放位置**：
> - 错判：`docs/finance-journal/postmortem/YYYY-MM-DD-<ticker>-<D类型>.md`
> - 成功：`docs/finance-journal/postmortem/YYYY-MM-DD-<ticker>-success.md`
>
> **强制铁律**：每次复盘必须写明"framework 如何修改"，写不出修改项 → 归因不到位 → 重写。

---

## 模板 A — 错判归因（通用）

```markdown
# 错判归因：<标的> <错判类型>  YYYY-MM-DD

## 0. 复盘摘要

| 字段 | 值 |
|------|---|
| 原决策日期 | YYYY-MM-DD |
| 原决策动作 | buy / add / trim / sell / hold / skip / ipo |
| 错判类型 | D1 / D2 / D3 / D4 / D5（必填） |
| 损失幅度 | -XX%（或机会成本） |
| 决策卡链接 | [原决策卡](../YYYY-MM-DD-<ticker>-<动作>.md) |
| 触发本次复盘 | 到期验证 / 主动触发 / 累计 3 次 |

---

## 1. 原决策摘要

- **当时判断**：<一句话>
- **核心假设**：<列 1-3 条>
- **预判（4 维）**：
  - T+3：...
  - T+6：...
  - T+12：...
  - T+24：...

---

## 2. 实际发生

- **实际路径**：<时间线>
- **T+X 实际值**：...
- **关键转折点**：...

---

## 3. 错判类型（单选，必填）

> 见 `self-improvement-loop.md` §L3 的 5 类定义

- [ ] **D1 数据源盲区**：关键信号被漏（SEC / HKEX / KOL / RSS 没抓）
- [ ] **D2 分析深度不够**：只读表层没到里层（MD&A / 2 跳推演 / 估值没算）
- [ ] **D3 预判不具体**：笼统废话无法证伪（"长期向好" / "估值合理"）
- [ ] **D4 执行纪律失控**：该执行没执行（止损没执行 / 追高 / 加仓亏损）
- [ ] **D5 认知盲点**：某类主题过度偏见（对周期股过度悲观 / 科技股过度乐观）

### 互斥判断顺序（优先判外因，最后才 D5）

```
Step 1: 是否漏了可公开获得的数据？→ 是 则 D1
Step 2: 是否读了但分析深度不够？→ 是 则 D2
Step 3: 预判是否具体、可证伪？→ 否 则 D3
Step 4: 是否该执行没执行（止损/仓位等）？→ 是 则 D4
Step 5: 以上都不是 → D5（最难改）
```

**不要直接跳 D5**：D5 是兜底，大多数错判属于 D1-D4（可补救）。

3. **为什么没注意到？** 因为 ...
4. **为什么当时觉得 OK？** 因为 ...
5. **根本原因是什么？** → ...

## 5. 如果重来，应该怎么做

- **数据源补**：要加哪些源 / 哪些订阅 / 哪些告警
- **分析步骤加**：决策树里应该多哪一步
- **Checklist 加条**：决策卡 / 论点卡里应该检查什么
- **执行规则加**：触发条件 / 止损刚性 / 加减仓节奏

## 8. 累计触发检查

> 填完此卡后，更新 `docs/finance-journal/INDEX.md` 错判统计表。

- 本次错判类型：D_
- 此类型累计次数：_（本次前 + 1）
- **是否累计 ≥ 3 次**：[ ] 是 → 强制触发 framework 升级 / [ ] 否 → 继续观察

### 如果累计 ≥ 3 次

触发词：**"framework 升级审查 D_"**

skill 会：
1. 列出该类型 3+ 次错判案例
2. 找共性 + 提炼通用升级方案
3. 引导用户修改对应 framework（填迭代日志）
4. commit 升级

---

## 诚实 Disclaimer

- 根因分析难免主观，允许 2 次返工
- D1-D5 归因边界模糊时选**更可补救的那一档**（D1 > D2 > D3 > D4 > D5）
- 累计 3 次触发升级只是阈值，严重单次错判也可单独升级
- 归因不是惩罚自己，是系统性进步的手段

**"错判不可避免，不归因才是真正的失败"**
```

```markdown
- **T+3 / T+6 / T+12 / T+24 实际值**：...

## 3. 成功归因（多选）

- [ ] **S1 数据深度**：挖到了别人没看到的信号
- [ ] **S2 分析深度**：推出了别人没推的链条
- [ ] **S3 预判精准**：预判的数字/时点接近实际
- [ ] **S4 执行纪律**：买入 / 加仓 / 减仓时点好
```

## 填充示例（错判归因）：NVDA 2023-Q1 踏空（D2 分析深度不够）

```markdown
# 错判归因：NVDA D2 分析深度不够  2026-04-23（回顾 2023-Q1 教训）

## 0. 复盘摘要

| 字段 | 值 |
|------|---|
| 原决策日期 | 2023-04-15 |
| 原决策动作 | skip（不买入） |
| 错判类型 | D2 分析深度不够 |
| 损失幅度 | 机会成本 +200%（NVDA 从 $280 → $830） |
| 决策卡链接 | [原 skip 决策](../2023-04-15-NVDA-skip.md) |
| 触发本次复盘 | 手动补录历史教训 |

## 1. 原决策摘要

- **当时判断**："Trailing PE 150x，太贵，Data Center 业务虽然增速 40%+ 但是短期已经 price in"
- **核心假设**：高 PE 必然压估值
- **预判**：T+6 估值回落到 PE 60-80，股价区间震荡
- **动作**：skip

## 2. 实际发生

- 2023-05 Q1 财报 Data Center 业务 Beat + Q2 guidance $11B（consensus $7.1B，+55%）
- 股价 $285 → $410（+44%）一个月内
- 2024 年 Data Center 营收超 $130B
- 股价涨到 $1200+（复权）

## 3. 错判类型

- [x] **D2 分析深度不够**

**互斥判断**：
- Step 1 D1？数据公开可查（SEC / 财报 / guidance），不是 D1
- Step 2 D2？✅ **只看 trailing PE，没读 Q2 guidance 里的 "HBM 供应锁死"** 这个 1 行字
- Step 3 D3？预判其实具体（PE 60-80）
- Step 4 D4？没实际交易，不是 D4
- 归因 → **D2**

## 4. 根因分析

1. 为什么错判？**只看当前 PE，没看 forward guidance 预期差**
2. 为什么出现？**当时还没沉淀"股市炒预期"铁律**
3. 为什么没注意到？**只看财报总结文章，没读电话会原文**
4. 为什么当时觉得 OK？**PE 150x 这种数字太"吓人"，让思维停止了深挖**
5. **根本原因**：停留在"表层 PE"层面，没穿透到里层的"HBM 供应 + Capex 超预期"

## 5. 如果重来

- **数据源补**：Bloomberg Terminal 电话会原文订阅，SeekingAlpha 深度解读
- **分析步骤加**：高 PE 股必须读 Q+1 guidance 再判断
- **Checklist 加条**：alpha-thesis-card 加"forward guidance vs consensus 差"字段
- **执行规则加**：PE 贵但 forward guidance 超预期 30%+ → 买小仓试错不是 skip

## 6. Framework 修改项

- [x] 在 `command-driven-analysis.md` §"股市是预期驱动" 加 NVDA 2023 案例（已完成 2026-04-22 压测）
- [x] `valuation-relative.md` §3 Step 4 强调"三情景 forward EPS" 而非 trailing PE
- [ ] `alpha-thesis-card.md` 加"forward guidance vs consensus" 字段（待做）
- [ ] 补 memory feedback：'PE 150x ≠ 泡沫，必须看 forward guidance vs consensus'

## 7. 教训

**教训**：PE 贵不是卖点，**guidance 超预期**才是决策依据。

## 8. 累计触发检查

- D2 累计：1 次（本次前 0 次）
- 暂未触发 framework 升级（< 3 次）
```

## 使用说明

1. **每次决策到验证时点**（T+3 / T+6 / T+12 / 终局）触发复盘
2. **复盘结论为"错"或"部分错"** → 写模板 A
3. **复盘结论为"成功"** → 写模板 B
4. **所有复盘必须写"Framework 修改项"** — 没有就是归因失败
5. **填完后更新 INDEX.md** 的错判统计表 + 待验证队列

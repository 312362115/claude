# Self-Improvement Loop（自我迭代闭环）

> skill 从"工具"进化成"真正分析师"的核心机制。
>
> **没有这套闭环 = skill 永远不会变强**。每次命题都从零开始，不积累经验、不修正盲点、不沉淀教训。

---

## 核心问题

人类顶级分析师为什么越做越强？

不是聪明，是**闭环**：
- 每次决策留档
- 事后严格验证
- 错判归因到方法论
- 下次更准

skill 要做到这点，需要系统性闭环。**4 层机制缺一不可**。

```
决策 → 留档 → 事后验证 → 错判归因 → Framework 升级 → 下次更准
  ↑                                                          ↓
  └──────────────────────────────────────────────────────────┘
```

---

## 4 层机制

### L1 决策留档（纪律层）

**要求**：**每次**命题分析产出的决策卡，**必须**归档到本地。

**位置**：`docs/finance-journal/YYYY-MM-DD-<ticker>-<动作>.md`

**动作分类**：
- `buy` / `add` / `sell` / `trim` / `hold` / `watchlist-add` / `skip`

**skip** 也要记录（决定"不做"也是决策，避免"其实我想过"的事后修正记忆）。

**强制字段**（决策卡 template 已包含）：
- 当前价位 / 目标价 / 止损
- 仓位
- 4 维预判（T+3 / T+6 / T+12 / T+24）
- 关键假设 + bear case
- 预期差强度自评

**铁律**：**没有决策卡 = 没做过决策**。不允许 "当时我就觉得 ..."。

---

### L2 事后验证（触发层）

**要求**：每张决策卡到预判时点自动触发复盘。

#### 验证日程（决策卡内置字段）

每张决策卡顶部必有：

```markdown
## 验证日程
- [ ] T+3 复盘日期：YYYY-MM-DD（回看 T+3 预判）
- [ ] T+6 复盘日期：YYYY-MM-DD
- [ ] T+12 复盘日期：YYYY-MM-DD
- [ ] 终局复盘日期：平仓后立即
```

#### 触发方式（手动为主）

用户主动调 skill 触发词：
- **"复盘持仓"** → 扫所有未验证决策卡 + 拉最新数据 + 生成复盘
- **"复盘 NVDA"** → 单股复盘
- **"verify T+3"** / **"verify T+6"** → 按时点筛选

skill 扫描 `docs/finance-journal/`，找未验证项 + 到期项，批量生成复盘报告。

#### 用户自动化（可选）

用户可自己配 launchd / cron，定时触发：
```bash
# 每周日跑一次
0 20 * * 0 claude-code "复盘本周所有到期决策"
```

skill 不主推自动化（见 command-driven-analysis.md "不做定时" 原则），但不阻止。

#### 验证报告格式

每次复盘输出 `docs/finance-journal/YYYY-MM-DD-<ticker>-verify-Tx.md`：

```markdown
# 复盘 - <Ticker> T+X 验证

## 原决策（链接到原卡）
日期：YYYY-MM-DD
动作：buy @ $XX

## 预判 vs 实际

### T+3 预判
- 预判：下季营收 $XXB，Data Center YoY +XX%
- 实际：$XX（±X%）
- 对错：✅ / ❌ / 部分

### 关键假设
- 假设 1：...（仍成立 / 被证伪）

## 股价表现
- 当时买入价：$XX
- 当前：$XX（±X%）
- 目标 / 止损触发情况：...

## 论点状态
- [ ] 仍成立 → 持有
- [ ] 部分证伪 → 减仓
- [ ] 全部证伪 → 清仓

## 行动
- 下一步：...
```

### L3 错判归因（学习层）

**要求**：每次复盘结论为"错"或"部分错"时，必须归因到具体类型。

#### 5 类错判（归因标签）

| 类型 | 含义 | 举例 |
|------|------|------|
| **D1 数据源盲区** | 关键信号被漏 | rss 没抓 / SEC 漏读 / 某 KOL 提前说过但没关注 |
| **D2 分析深度不够** | 只读表层没到里层 | 只看 Beat/Miss 没读 MD&A / 没推 2 跳 / 没算估值 |
| **D3 预判不具体** | 笼统废话无法证伪 | "长期向好 / 看好 AI / 估值合理" 类 |
| **D4 执行纪律失控** | 该执行没执行 | 该止损不止损 / 追高 / 加仓亏损 |
| **D5 认知盲点** | 某类主题过度偏见 | 对周期股过度悲观 / 对科技股过度乐观 |

#### 错判归档位置

`docs/finance-journal/postmortem/YYYY-MM-DD-<ticker>-<错判类型>.md`

**template**：

```markdown
# 错判归因 - <标的> <错判类型>

## 原决策
- 决策日：YYYY-MM-DD
- 决策内容：<链接到决策卡>

## 错在哪
- 预判：...
- 实际：...

## 归因类型
- [ ] D1 数据源盲区
- [ ] D2 分析深度不够
- [ ] D3 预判不具体
- [ ] D4 执行纪律失控
- [ ] D5 认知盲点

## 根因分析（5 Why）
1. 为什么错判？因为 ...
2. 为什么出现这个？因为 ...
3. 为什么没注意到？因为 ...
4. 为什么当时觉得 OK？因为 ...
5. 根因是什么？→ ...

## 如果重来，应该怎么做
- 数据源补：...
- 分析步骤加：...
- Checklist 加条：...

## 拟对 Framework 的修改
- [ ] 更新 `<framework>.md` 的 XX 章节
- [ ] 在 `<template>.md` 加 XX 检查项
- [ ] 在 blacklist 补 XX 源
- [ ] 补 memory 条目
```

#### 累积触发 Framework 升级

**规则**：
- 任一错判类型 ≥ **3 次** → 强制触发该类型对应 framework 升级
- 例：D1 数据源盲区累积 3 次 → 必须升级 data-sources 或 RSS 订阅清单
- 例：D3 预判不具体累积 3 次 → 必须强化 template 的可证伪性约束

---

### L4 Framework 升级（进化层）

**要求**：每个 framework / template 顶部必有"**迭代日志**"，记录每次升级 + 触发原因。

#### 迭代日志格式（在文件顶部 frontmatter 下方）

```markdown
## 版本迭代日志

| 版本 | 日期 | 变更 | 触发来源 |
|------|------|------|---------|
| v0.1.0 | 2026-04-20 | 初始版本 | - |
| v0.1.1 | 2026-06-15 | 补"大厂 Capex 连续下调 = 顶部信号" | 2026-06 NVDA 错判（D2）累积第 3 例 |
| v0.1.2 | 2026-08-20 | 修正"电力股入场时点" | 2026-08 CEG 追高教训（D4） |
```

#### 升级流程

```
错判归因累积 ≥ 3 次
  ↓
skill 触发词："framework 升级审查"
  ↓
列出该类错判的 3+ 案例
  ↓
用户决定：
  - 修改 framework 对应章节
  - 更新 checklist
  - 补 blacklist
  - 写 memory
  ↓
填迭代日志
  ↓
commit
```

#### 升级不是"改越多越好"

**反模式**（警惕）：
- ❌ 每次错判都改 framework → 会变成"过度拟合历史"
- ❌ 改 framework 后删除原版 → 丢失历史版本价值
- ❌ 个人化过强的教训硬写进 framework → 污染通用性

**好的升级**：
- ✅ 3+ 同类错判的共性 → 沉淀方法
- ✅ 单次严重错判（如爆仓） → 补铁律
- ✅ 行业结构性变化（如 2024 IEX Cloud 停服） → 更新腐烂标记

## 目录结构（支撑闭环）

**核心原则**：金融分析相关的所有产出全部集中到 `docs/finance-journal/` 一个目录，便于索引和定期巡视。

```
.claude/
├── skills/finance-analysis/
│   ├── references/frameworks/   ← 方法论（L4 升级点）
│   ├── references/templates/    ← 决策卡 / 复盘模板
│   └── references/integrations/ ← 数据对接
│
docs/finance-journal/            ← 所有金融分析产出集中在此
├── INDEX.md                      ← **决策索引**（每次操作后更新）
│
├── YYYY-MM-DD-<ticker>-<动作>.md          ← L1 决策留档（扁平）
├── YYYY-MM-DD-<ticker>-verify-T+N.md      ← L2 验证报告（扁平）
│
├── postmortem/                   ← L3 错判归因 + 成功决策复盘（统一放这里）
│   └── YYYY-MM-DD-<ticker>-<D类型>.md
│
├── research/                     ← 命题分析报告（深度研究 / 未决策的命题产出）
│   └── YYYY-MM-DD-<ticker>-<分析类型>.md
│
├── monthly/                      ← 月度复盘汇总
│   └── YYYY-MM.md
│
└── quarterly/                    ← 季度 / 年度命中率报告
    └── YYYY-QN.md

docs/strategies/                  ← 沉淀的套路 + 认知升级（跨标的通用方法论）
    └── ...

~/.claude/projects/-Users-.../memory/  ← 跨会话 feedback
    └── ...
```

### 文件归属规则

| 产出类型 | 放哪 | 文件名 |
|---------|------|-------|
| 决策卡（buy / sell / skip 等） | 根目录扁平 | `YYYY-MM-DD-<ticker>-<动作>.md` |
| 验证报告（T+3 / T+6 / 终局） | 根目录扁平 | `YYYY-MM-DD-<ticker>-verify-T+N.md` |
| 错判归因 | `postmortem/` | `YYYY-MM-DD-<ticker>-<D类型>.md` |
| 成功经验复盘 | `postmortem/` | `YYYY-MM-DD-<ticker>-success.md` |
| 命题分析（未产出决策） | `research/` | `YYYY-MM-DD-<ticker>-analysis.md` |
| 月度汇总 | `monthly/` | `YYYY-MM.md` |
| 季度 / 年度 | `quarterly/` | `YYYY-QN.md` |

**为什么集中在一个目录**：
- 方便周 / 月 / 季度巡视（一个目录看全）
- INDEX.md 作为统一入口
- 错判归因和决策卡相邻，归因时容易回看原决策
- 不和 `docs/decisions/`（CLAUDE.md 规范的"项目复盘"目录）混淆

## Skill 触发词（支持闭环）

在 SKILL.md 命题驱动路由之外，新增"**复盘类命题**"：

| 触发词 | 动作 |
|-------|------|
| "复盘 / 回看 / verify / 业绩验证" | 批量扫 finance-journal 到期决策 |
| "复盘 <ticker>" | 单股复盘 |
| "错判归因" | 生成 postmortem 模板 |
| "framework 升级审查" | 列出累积错判，引导升级 |
| "命中率统计" | 统计过去 N 月决策命中率 |

这些触发后 skill 会读 `docs/finance-journal/`（含 postmortem 子目录）做分析，**和命题分析是两种不同模式**。

## 命中率统计（长期 KPI）

每季度 / 每年跑一次，衡量 skill 是否在进步：

```markdown
# 2026-Q2 命中率报告

## 总体
- 决策总数：XX
- T+3 预判命中率：XX%
- T+6 预判命中率：XX%
- 胜率（盈利交易 / 总交易）：XX%
- 盈亏比：X:1

## 分类型
- 套路 A 个股分析命中率：XX%
- 套路 B 政策事件：XX%
- 套路 C 事件驱动：XX%
- 套路 D 主题赛道：XX%

## 错判分布
- D1 数据源盲区：X 次
- D2 分析深度不够：X 次
- D3 预判不具体：X 次
- D4 执行纪律失控：X 次
- D5 认知盲点：X 次

## 本季 Framework 升级
- 升级 1：... （触发来源）
- 升级 2：...

## 下季改进重点
- ...
```

**目标（长期 KPI）**：
- T+3 预判命中率 > 60%（低于此 → 方法论有问题）
- 盈亏比 > 2:1（赢比输多一倍以上）
- 最大单标的亏损 < -20%（止损纪律）

## 冷启动注意事项

**前 3 个月的 skill 不会有命中率数据**，正常。

冷启动期做：
1. **每次命题都填完整决策卡**（哪怕是模拟盘）
2. **预判必须可证伪**（即使最终没交易）
3. **小额实盘** 验证闭环能跑通
4. **不追求命中率**，先追求**流程完整**

3 个月后开始有数据。6 个月后第一次命中率评估。

## 和 Self-Improvement 对抗的反模式

避免陷阱：

| 反模式 | 表现 | 后果 |
|--------|-----|------|
| 只留档成功决策 | 选择性记忆，只写赚钱的 | 命中率虚高，实际没进步 |
| 预判改动但不留痕 | 事后改预判迎合结果 | 闭环断裂 |
| 错判不归因（"运气不好"） | D5 认知盲点无限累积 | Framework 永远不升级 |
| 归因到外因（"大盘暴跌"） | 失去自我反思机会 | 下次同样错 |
| 复盘报告写完就算 | 不真正改 framework | 4 层降为 2 层 |

**最严格自检**：**每个错判归因都能写出具体的 framework 修改项吗？** 写不出来 → 归因不到位。

## 诚实 disclaimer

- 闭环**需要用户配合执行**。用户不填决策卡、不复盘、不归因 → 这套机制形同虚设
- skill 提供方法，不强制纪律
- 3-6 个月才能看到效果，前期是"纯投入"
- **这才是 skill 从 "工具" 到 "分析师" 的分水岭**

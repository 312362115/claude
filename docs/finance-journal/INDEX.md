# Finance Journal Index

> 金融决策记录索引。所有投资决策卡 / 验证报告 / 错判归因的入口。
>
> **维护规则**：新建决策、状态变更、完成验证、做完归因 → 立即更新本文件。
>
> **关联 skill**：`finance-analysis` skill 的 self-improvement-loop framework。

## 状态标记

| 标记 | 含义 |
|------|------|
| `[?]` | watchlist — 观察中，未入场 |
| `[~]` | active — 持仓中 |
| `[x]` | closed — 已平仓，正收益退出 |
| `[!]` | stopped-out — 触发止损退出 |
| `[-]` | skip — 决定不做（也要留档） |
| `~~...~~` | dropped — 决策取消 |

## 🛡️ 守护检查（每次 INDEX 更新时触发）

> **INDEX 不是被动存储，是闭环守护节点**。每次更新都必须运行 4 类扫描，主动提醒用户未完成的动作。
>
> 这是防止"决策留档了但没跟到底"的关键机制——避免 self-improvement-loop 形同虚设。

### 扫描 1：到期未验证（阻断性）

```
遍历"待验证队列"
  └─ 当前日期 ≥ 验证日程日期？
      ├─ 是 且未完成验证 → 🔴 红色警报，必须优先处理
      ├─ 将在 3 天内到期 → 🟡 黄色提醒
      └─ 未到期 → 静默
```

**输出给用户**：
- 列出所有🔴 红色警报项（到期未验证的决策）
- 建议：先喊"复盘 <ticker>"或"复盘持仓"清理队列
- **在所有 🔴 警报处理完之前，不推荐做新决策**

### 扫描 2：D 类型累计阈值（触发升级）

```
读取"按错判类型统计"表
  └─ 任一类型累计次数 ≥ 3？
      ├─ 是且未升级过 → 🔴 强制触发 framework 升级审查
      └─ 否 → 更新累计次数即可
```

**输出给用户**：
- 列出累计 ≥ 3 次的错判类型 + 对应 3 个案例链接
- 建议：喊"framework 升级审查 D_" 进入升级流程
- **不升级的话下次同类错判会继续累积**

### 扫描 3：观察池过期（防止"永远在观察"）

```
遍历"当前持仓 / 观察"下的 [?] watchlist 项
  └─ watchlist 添加日期距今 > 90 天？
      ├─ 是 → 🟡 过期提醒，要求用户决策
      └─ 否 → 静默
```

**输出给用户**：
- 列出观察 > 90 天的项目
- 选项：(a) 升级为 active 开仓 / (b) 转 skip 存档 / (c) 继续观察但刷新理由
- 防止观察池变成"决策黑洞"

### 扫描 4：持仓红线（防止"该止损没止损"）

```
遍历"当前持仓 / 观察"下的 [~] active 项
  └─ 读取决策卡中的止损价 + 目标价
      ├─ 现价 距止损 < 5% → 🔴 接近止损线
      ├─ 现价 ≥ 目标价 80% → 🟢 接近目标（考虑减仓）
      ├─ 持仓时间 > 预设时间维度 → 🟡 过持有期
      └─ 均正常 → 静默
```

**输出给用户**：
- 接近止损 → 是否止损？
- 接近目标 → 是否部分减仓锁利润？
- 超过持有期 → 论点是否还成立？

### 守护检查报告（每次 INDEX 更新后输出）

```markdown

### 🔴 阻断性（优先处理）

- X 条即将到期验证：...

### 🟢 机会性（可选处理）

1. [最关键动作]
2. ...
```

## 维护指引

### 什么时候更新本 INDEX

| 事件 | 更新位置 | 同时触发的守护检查 |
|------|---------|------------------|
| 新建决策卡 | "当前持仓 / 观察" + "按时间归档" | 4 类全扫 |
| 决策状态变更（active → closed 等） | 改状态标记 | 4 类全扫 |
| 完成 T+3 / T+6 / T+12 / 终局 验证 | "待验证队列"移除 + 原卡勾选 | 扫 1 + 2 |
| 发生错判并归因 | "错判归因清单"追加 + 累计次数 +1 | 扫 2（触发升级？） |
| 累计 3 次同类错判 | 触发 framework 升级提醒 | — |
| 月底 | 生成 "月度复盘"并加入索引 | 4 类全扫 |
| 季度末 / 年末 | 生成 "季度 / 年度报告" | 4 类全扫 |

**铁律**：每次 INDEX 更新 = 必跑守护检查。守护检查报告 append 到该次更新的 commit message 或展示给用户。

### 触发词对应动作

| skill 触发词 | INDEX 操作 | 守护检查 |
|-------------|----------|---------|
| "复盘持仓" | 扫"待验证队列" → 批量生成验证 | 扫 1 + 2 |
| "复盘 <ticker>" | 按标的搜本 INDEX → 加载对应卡 + 数据 | 扫 1 + 4 |
| "错判归因" | 追加"错判归因清单" + 累计次数 | 扫 2 |
| "framework 升级审查" | 检查累计次数表，任一 ≥ 3 列出对应错判案例 | 扫 2 |
| "命中率统计" | 从本 INDEX + 月度文件汇总计算 | 4 类全扫 |
| **"巡检"** / **"扫一下"** | 专门触发 4 类守护检查，无其他动作 | 4 类全扫 |

### 目录结构（所有金融分析产出集中在此）

```
docs/finance-journal/
├── INDEX.md                            ← 本文件（统一索引）
│
├── YYYY-MM-DD-<ticker>-<动作>.md        ← L1 决策卡（扁平）
├── YYYY-MM-DD-<ticker>-verify-T+N.md   ← L2 验证报告（扁平）
│
├── postmortem/                         ← L3 错判归因 + 成功经验复盘
│   ├── YYYY-MM-DD-<ticker>-<D类型>.md       # 错判
│   └── YYYY-MM-DD-<ticker>-success.md       # 成功经验
│
├── research/                           ← 命题分析报告（未产出决策 / 深度研究）
│   └── YYYY-MM-DD-<ticker>-analysis.md
│
├── monthly/                            ← 月度汇总
│   └── YYYY-MM.md
│
└── quarterly/                          ← 季度 / 年度命中率
    └── YYYY-QN.md
```

**原则**：金融分析相关所有产出全部集中在 `docs/finance-journal/`，不分散到 `docs/decisions/`（后者是 CLAUDE.md 的项目开发复盘目录）。

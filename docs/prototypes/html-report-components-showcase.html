    一、基础排版组件

    组件 #1

    Report Header — 报告头

    用途：报告标题页，展示标题、副标题和元数据（日期、类型、深度等）。
    CSS 类：.report-header .report-title .report-subtitle .report-meta .meta-item
    MD Preview 适用性：可选 — MD 文件通常没有元数据区，但可用于渲染 YAML frontmatter

    上方页面顶部就是 Report Header 的实际效果。包含：

    • 标题：2.1rem, weight 800, 负字间距，视觉冲击力强
    • 副标题：1.05rem, 次要色，补充说明
    • 元数据条：圆角背景卡片，flex 布局，蓝点分隔符
    组件 #2

    Typography — 排版体系

    用途：正文排版基础 — 标题层级、段落、列表、链接、加粗。
    CSS 类：h2 h3 h4 p ul/ol a strong
    MD Preview 适用性：核心 — 这是 MD 渲染的基础

    H2 标题 — 章节级（1.45rem / 700）

    H2 有底部 1px 浅色分割线，上方 56px 间距营造"呼吸感"。适合 MD 中的 ## 标题。

    H3 标题 — 小节级（1.12rem / 600）

    H3 无底线，更紧凑。适合 MD 中的 ### 标题。

    H4 标题 — 段内标题（0.95rem / 600 / 次要色）

    H4 使用次要文字色，视觉层级更弱。适合 MD 中的 #### 标题。

    行内元素：加粗文字、超链接（hover 显示下划线）、行内代码。

    三、专业组件（报告专属）

    组件 #11

    Executive Summary — 执行摘要

    用途：报告开头的关键结论总结，渐变背景突出重要性。
    CSS 类：.exec-summary
    MD Preview 适用性：报告专属 — 不下沉到通用 MD 预览

    Executive Summary

    本次调研覆盖了 15 个技术方案，经过三轮筛选和 PoC 验证，核心结论如下：

    • 方案 A（Redis Session）综合评分最高，推荐采用
    • 性能瓶颈主要集中在序列化环节，p99 延迟可控制在 2ms 以内
    • 合规要求可在 Q1 结束前满足，无阻塞风险
    组件 #12

    Callout — 提示框（五种类型）

    用途：突出显示提示、建议、重要、警告、危险信息。对应 GitHub Alerts 五种类型。
    CSS 类：.callout + .callout-note / .callout-tip / .callout-important / .callout-warning / .callout-caution
    MD 语法：> [!NOTE] / > [!TIP] / > [!IMPORTANT] / > [!WARNING] / > [!CAUTION]
    MD Preview 适用性：核心 — GitHub Flavored Markdown 标准语法
    Note

    补充说明信息，帮助用户更好地理解上下文。

    Tip

    实用建议，帮助用户更高效地完成任务或获得更好的结果。

    Important

    关键信息，用户需要了解才能顺利进行。不可忽略的重要事项。

    Warning

    需要注意的风险或限制条件，忽略可能导致问题。

    Caution

    严重风险或破坏性操作警告，执行前务必再三确认。

    组件 #13

    Card Grid — 卡片网格

    用途：并列展示方案对比、关键发现、功能特性等。
    CSS 类：.card-grid .card .card-tag + .tag-*
    MD Preview 适用性：报告专属 — 需要特定 HTML 结构，MD 语法无法直接表达
    推荐

    Redis Session

    成熟稳定，撤销简单（删 key 即失效），已有集群可复用，运维成本低。

    备选

    JWT + 黑名单

    无状态优势明显，但撤销场景需额外维护黑名单，本质上又引入了状态。

    已排除

    Cookie Session

    现有方案，无法满足主动撤销的合规要求，不支持水平扩展。

    组件 #15

    VS Block — 对比块

    用途：方案 A vs B 双栏对比，顶部色条 + 图标区分两侧。
    CSS 类：.vs-block .vs-col .vs-label + .label-a/b .vs-icon
    MD Preview 适用性：报告专属 — 需要特定 HTML 结构
    A Redis Session（推荐）
    • 删 key 即撤销，确定性高
    • 已有集群，零额外基础设施成本
    • p99 延迟 < 2ms
    • 运维团队熟悉
    B JWT + 黑名单（备选）
    • 无状态架构，水平扩展性好
    • 客户端持有 token，减少服务端查询
    • 撤销需维护黑名单，本质引入状态
    • token 泄露风险窗口大
    组件 #16

    Timeline — 时间轴

    用途：技术演进路线、项目里程碑、历史事件梳理。
    CSS 类：.timeline .tl-item .tl-dot .tl-title .tl-desc
    MD Preview 适用性：报告专属 — 需要特定 HTML 结构
    Phase 1：基础改造 Week 1-2
    Redis 连接模块封装，session 中间件替换，基本功能验证。
    Phase 2：安全加固 Week 3
    Session 加密、主动撤销机制、异常检测与告警。
    Phase 3：灰度上线 Week 4
    10% 流量灰度，监控 Redis 负载和 session 命中率，逐步放量。
    Phase 4：全量 & 复盘 Week 5
    全量切换，下线旧 session 逻辑，合规审查复检，经验沉淀。
    组件 #17

    Badges — 徽章体系

    用途：行内状态标签（表格/标题中的快速标注）和来源层级标注。
    CSS 类：行内徽章 .badge + .badge-*；来源层级 .tier-badge + .tier-t1/t2/t3
    MD Preview 适用性：来源层级为报告专属；行内徽章可选

    行内徽章（Inline Badge）

    推荐 已确认 待定 实验性 已废弃

    圆角药丸形，0.72rem 字号，适合嵌入标题或表格单元格中做快速标注。

    来源层级徽章（Tier Badge）

    一手来源 T1 — 二手来源 T2 — 三手来源 T3

    上标定位（vertical-align: super），0.65rem 微型徽章，用于标注引用来源的可信度层级。

    假设验证状态

    已验证 — 已证伪 — 证据不足

    组件 #18

    Sidebar + TOC — 侧边目录

    用途：固定侧边栏，自动提取 h2/h3 生成目录，滚动高亮当前章节。
    CSS 类：.sidebar .toc-list .toc-item .depth-3 .active
    MD Preview 适用性：核心 — preview-md 已有此功能，样式需同步

    左侧 260px 固定侧边栏就是实际效果。特点：

    • h2 正常缩进，h3 额外 padding-left: 36px + 更小字号
    • 当前激活项：蓝色左边框 + 浅蓝背景 + 加粗
    • Hover 效果：蓝色文字 + 浅蓝背景
    • 窄滚动条（4px）匹配整体精致感
    • JS 自动生成目录 + requestAnimationFrame 滚动监听

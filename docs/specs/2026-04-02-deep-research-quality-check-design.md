# Deep-Research 报告质量自动检测

## 背景与动机

基于 2026-03-30 综合评估，deep-research skill 报告质量评分 7.9，目标提升到 9。当前 SKILL.md 在各步骤中嵌入了质量要求（来源分级、假设验证、反面观点等），但没有统一的自动检测环节——质量全靠 AI 自觉遵守，缺漏难以发现。

- **提出方**：backlog P0 项，质量优化的基础设施
- **约束**：纯 prompt 驱动，不引入外部脚本；检查结果在终端展示，不污染报告内容

## 现状分析

- SKILL.md Step 1-5 各步骤有散落的质量要求（T1/T2/T3 标记、假设验证表、反面观点章节等）
- 没有集中的自检环节，报告写完即输出
- 常见质量问题：引用缺少层级标记、假设验证表不完整、核心结论缺少高可信来源支撑
- 不改后果：质量依赖随机性，报告水平波动大

## 调研与备选方案

### 方案 A：外部脚本检测

- **调研内容**：用 Python 脚本解析 Markdown/HTML 报告，正则匹配检查各项指标
- **验证结果**：可行，但引入脚本维护成本，且无法理解语义（如"结论是否有证据支撑"需要语义理解）
- **结论**：过重，检查维度中有语义判断需求，纯正则不够

### 方案 B：纯 prompt 自检（选定）

- **调研内容**：在 SKILL.md 中新增 Step 6，让 AI 在报告写完后按检查清单逐项自检
- **验证结果**：AI 擅长结构化检查和语义理解，且零维护成本
- **结论**：最简方案，覆盖所有检查维度，与现有流程无缝衔接

### 方案 C：独立 skill

- **调研内容**：将质量检测做成独立 skill，可复用于其他报告类场景
- **验证结果**：当前只有 deep-research 需要，独立 skill 过度设计
- **结论**：不必要的抽象，等有第二个使用场景再考虑

## 决策与取舍

采用方案 B（纯 prompt 自检）。

- **核心理由**：零依赖、零维护，AI 本身就能做语义级检查，与现有 SKILL.md 流程自然衔接
- **取舍说明**：放弃脚本的确定性（脚本每次结果一致），换取灵活性和语义理解能力
- **风险**：AI 自检可能有盲区（自己写的报告自己查），但 4 个检查维度都是结构化规则，盲区有限

## 技术方案

### 新增位置

在 SKILL.md 的 Step 5（报告写作）之后新增 **Step 6：质量自检**。

### 检查维度

#### 1. 结构完整性

检查报告是否包含当前研究类型模板要求的所有必备章节：

| 通用必备章节 | 说明 |
|------------|------|
| 执行摘要 | ≤500 字，含核心结论 + 关键发现 + 行动建议 |
| 现状基线 | 内部现状 + 对比基线 |
| 反面观点与回应 | 每个核心结论的反面论证 |
| 假设验证结果 | 假设 → 结果 → 证据表 |
| 核心结论 | 带数据和来源层级标注 |
| 行动建议 | 可执行的下一步 |
| 参考资料 | 按 T1/T2/T3 分组 |

**处理方式**：不可自动修复。缺失章节列出，提示用户是否补写。

#### 2. 引用格式

检查规则：
- 参考资料是否按 T1/T2/T3 三级分组（"一手来源 [T1]"、"二手来源 [T2]"、"三手来源 [T3]"）
- 每条引用是否有 `[T1]`/`[T2]`/`[T3]` 标记
- 链接格式是否为 `[标题](URL) — 引用说明`
- 正文中引用数据时是否标注了来源层级

**处理方式**：可自动修复。格式不符的自动补齐标记和分组。

#### 3. 假设验证表

检查规则：
- 报告中是否存在假设验证表
- 每个初始假设（Step 1.4 中提出的）是否都在表中
- 每行是否包含：假设内容、验证结果（✅ 已验证 / ❌ 已证伪 / ⚠️ 证据不足）、支撑证据及来源层级

**处理方式**：
- 表存在但格式不完整 → 自动修复格式
- 表不存在 → 提示用户，因为需要回顾全文生成内容

#### 4. 核心结论来源支撑

检查规则（沿用 SKILL.md 现有要求）：
- 每个核心结论是否满足"1+ T1 来源 **或** 2+ 独立 T2 来源"
- 仅有 T3 支撑的结论标记为不达标

**处理方式**：不可自动修复。列出不达标结论及当前来源情况，提示用户是否补充搜索。

### 执行流程

```
报告写完
  ↓
读取报告全文
  ↓
逐项检查 4 个维度
  ↓
自动修复可修项（引用格式、验证表格式）
  ↓
终端输出检查摘要
  ↓
有不可修项？→ 列出问题，询问用户是否处理
  ↓
用户确认 → 补充/跳过 → 完成
```

### 终端输出格式

```
📋 报告质量自检
─────────────────────────────────────
✅ 结构完整性：7/7 必备章节齐全
🔧 引用格式：修复了 3 处缺失的来源层级标记
✅ 假设验证表：4 个假设均有验证结果和证据来源
⚠️ 核心结论来源支撑：
   · "市场规模将在2027年达到500亿" → 仅 1 个 T3 来源，不达标
   · "技术 A 性能优于技术 B" → 2 个独立 T2 来源，达标

总计：3 项通过，1 项需处理
─────────────────────────────────────
是否对不达标项补充搜索？(y/n)
```

### 不涉及的内容

- 不增加 T1/T2 整体比例要求（沿用现有"核心结论必须有 T1/T2 支撑"规则）
- 不生成独立的质量检查文件
- 不引入外部脚本或依赖

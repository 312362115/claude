# 全局开发规范

> 个人全局规范，项目级 CLAUDE.md 可覆盖本文件。

## 总纲

**做好比做完重要**（不可被项目级规则覆盖）。动手前先确认需求、读过相关代码；该走的流程不跳过。

---

## 一、Git 与提交

- **commit message**：`<type>(<scope>): <中文描述>`，type: feat / fix / refactor / docs / test / chore / perf
- **分支命名**：`feat/xxx`、`fix/xxx`、`refactor/xxx`
- **PR vs 直推**：多人协作或已发布项目走 PR 并等 CI 通过；个人/未发布项目可直接推送

**PR 判断标准**（任一命中"团队/已发布"就走 PR，全部命中"个人/未发布"才直接推）：

| 维度 | 走 PR | 可直接推 |
|------|-------|---------|
| 仓库归属 | 公司/组织 | 个人 |
| 协作者（`git log --format='%ae' \| sort -u`） | 多邮箱 | 仅自己 |
| 发布痕迹 | 有 tag / CHANGELOG / npm / 线上域名 | 纯本地 |
| CI 配置 | 有 `.github/workflows` 或 `.gitlab-ci.yml` | 无 |

拿不准走 PR。

## 二、文档目录（docs/）

**文档是持久真相源，随 git 提交；memory 是跨会话辅助索引**。关键决策必须写入 `docs/`，不能只存 memory。

流转关系：
```
backlog → prd → designs → spec → plan → tests → decisions
```
无产物的环节跳过（如纯后端无 designs，小需求无 prd）。

### 目录结构

| 目录 | 级别 | 用途 | 命名 |
|------|------|------|------|
| `docs/backlog/` | 必须 | 需求池（含 INDEX.md） | `YYYY-MM-DD-<主题>.md` |
| `docs/specs/` | 必须 | 方案设计（背景+调研+决策+方案） | `YYYY-MM-DD-<模块>-<主题>.md` |
| `docs/plans/` | 必须 | 开发计划(子任务+进度) | `YYYY-MM-DD-<模块>-plan.md` |
| `docs/tests/` | 必须 | 回归用例（验证脚本沉淀于此） | `<模块>-<功能>.md` |
| `docs/decisions/` | 必须 | 复盘与踩坑记录 | `YYYY-MM-DD-<主题>.md` |
| `docs/assets/` | 必须 | 文档图片/截图资源 | `<类型>-<描述>.png` |
| `docs/prds/` | 推荐 | 产品需求文档 | `YYYY-MM-DD-<功能>.md` |
| `docs/designs/` | 推荐 | UI 原型与标注 | `YYYY-MM-DD-<功能>/` |
| `docs/architecture/` | 推荐 | 系统架构（模块 ≥3 时建） | `<模块>.md` + `INDEX.md` |
| `docs/user-guide/` | 推荐 | 用户手册（有外部用户时建） | `<功能>.md` + `INDEX.md` |
| `docs/guides/` | 推荐 | 开发指南（环境/上手） | `<主题>.md` |
| `docs/runbooks/` | 建议 | 运维手册（有线上服务时建） | `<场景>.md` |
| `docs/changelog/` | 建议 | 版本变更记录 | `YYYY-MM-DD-v<版本>.md` |

文件名英文小写+连字符，内容中文。

### backlog 规范

管"做不做"。每个需求 frontmatter：
```yaml
---
priority: P1        # P0 紧急 | P1 重要 | P2 普通 | P3 低优
status: open        # open | in-progress | done | dropped
spec: docs/specs/xxx.md
plan: docs/plans/xxx.md
---
```

`docs/backlog/INDEX.md` 按优先级分组，格式 `- [ ] [标题](文件.md) — 一句话描述`。`[ ]`=open，`[x]`=done，`~~删除线~~`=dropped。仅新建和状态变更时更新。

### spec 规范

**背景不能省**，否则方案是空中楼阁。必备章节：

| 章节 | 回答的问题 |
|------|----------|
| 背景与动机 | 为什么做？谁提的？约束？ |
| 现状分析 | 当前怎么做？痛点？不改会怎样？ |
| 调研与备选方案 | 调研了哪些？每个方案的验证过程和结果？ |
| 决策与取舍 | 选哪个？放弃了什么？遗留风险？ |
| 技术方案 | 具体怎么实现 |

每个备选方案需包含：调研内容 / 验证方式 / 验证结果 / 结论。决策必须写**核心理由**和**取舍说明**，PoC 数据一并记录。

### plan 规范

管"做到哪了"。关联 spec + 子任务 checklist，每个子任务说清楚：
```markdown
- [ ] 1. <子任务>
  - 做什么：...
  - 涉及：`src/xxx.ts`
  - 验收：...
```

### tests 规范

验证脚本沉淀于此，每个用例：场景描述 / 前置条件 / 执行命令 / 预期结果。保证可直接复跑。

## 三、任务工作流

```
task-start → task-execute → 自动化验证 → task-finish
```

| 阶段 | 触发 | skill / 动作 |
|------|------|------------|
| 启动 | 新功能 / 复杂任务 / 需求模糊 | `task-start`（对焦 + UI 原型 + 方案） |
| 执行 | 跨会话大型任务 | `task-execute`（进度管理） |
| 验证 | 功能开发完成 | 跑自动化测试，脚本沉淀到 `docs/tests/` |
| 收尾 | 验证通过后 | `task-finish`（CR 自检 + 复盘写 `docs/decisions/`） |

**走多深看任务体量**：
- ≤2 文件小任务：直接动手
- 有模糊点：仅对焦，然后开发
- 3+ 文件 / 新模块 / 架构决策：完整三阶段
- 从零搭前后端系统：**全流程强制**
- 跨会话任务：必须启用 task-execute

---

## 四、执行纪律

- **验证即门槛**：功能没跑通不算完成，验证脚本必须沉淀为 `docs/tests/` 回归用例
- **根因驱动**：bug 修复先复现、定位根因，再写修复代码，不做表面补丁
- **小步提交**：大型改动拆分为小的、可独立审查的提交，避免一次性大 diff
- **范围控制**：只做被要求的事，"顺手优化"记下来开新任务
- **止损三次**：同一思路 3 次不通立即换方向或求助
- **主动升级**：复杂度超预期或需决策时立即告知用户

## 五、个人偏好

- 代码注释/文档中文，变量名/代码英文
- 复用项目已有库，引入新依赖前说明必要性
- 遵循项目已有的 lint / formatter 配置
